Projet Personnel M2L

Refonte d'une application Java mono-utilisateur en solution multi-utilisateurs avec base de données MySQL, authentification sécurisée et trois niveaux d'habilitation distincts.

Contexte du projet

Ce projet a été réalisé dans le cadre de mon BTS SIO. La Maison des Ligues de Lorraine (M2L) utilise une application pour gérer les employés des ligues sportives. L'objectif était de transformer une application Java en ligne de commande mono-utilisateur (utilisant la sérialisation fichier) en une solution robuste, multi-utilisateurs, avec base de données centralisée.

Technologies : Java 8, MySQL 8, JDBC, BCrypt, Maven

Architecture : 3-tiers (Présentation → Métier → Données)

Repo GitHub : github.com/ImOriane

🎭 Les 3 niveaux d'habilitation

Employé Simple
  • ✓ Consulter l'annuaire des employés
  • ✓ Voir les ligues existantes
  • ✗ Aucun droit d'écriture
Administrateur de Ligue
  • ✓ Tous les droits employé
  • ✓ Gérer les employés de sa ligue
  • ✓ Ajouter/modifier/supprimer
  • ✗ Limité à sa ligue uniquement
Super-Administrateur
  • ✓ Accès à tous les employés
  • ✓ Gérer les administrateurs
  • ✓ Créer/supprimer des ligues
  • ✓ Compte root unique

🏗️ Architecture 3-tiers

💻

Couche Présentation (CLI)

Interface en ligne de commande gérée par CommandLineMenus. Les menus s'adaptent dynamiquement selon le rôle de l'utilisateur connecté.

⚙️

Couche Métier

Classes GestionPersonnel, Ligue et Employe encapsulant les règles métier : gestion des rôles, contrôle des droits, validation des données.

🗄️

Couche Données (JDBC)

Classe JDBC implémentant l'interface Passerelle. Requêtes préparées (PreparedStatement) pour éviter les injections SQL.

📊 Base de données

Schéma relationnel

ligue TABLE
├── id_ligue INT AUTO_INCREMENT PRIMARY KEY
└── nom VARCHAR(100)

personnels TABLE
├── id_personnel_ INT AUTO_INCREMENT PRIMARY KEY
├── nom_perso VARCHAR(100)
├── prenom_perso VARCHAR(100)
├── mail_perso VARCHAR(150) UNIQUE
├── role_perso ENUM('root','admin','utilisateur')
├── password_perso VARCHAR(255)  -- BCrypt hash
├── date_arrivée DATE
├── date_depart_ DATE NULL
└── id_ligue INT FOREIGN KEY → ligue(id_ligue)

Contraintes : Clés étrangères, intégrité référentielle, unicité des emails, valeurs ENUM pour les rôles.

🔒 Sécurité & Authentification

  • Hashage BCrypt : Mots de passe stockés avec jbcrypt 0.4, vérification par BCrypt.checkpw()
  • Contrôle d'accès : Vérification des droits dans chaque action, gestion de l'exception DroitsInsuffisants
  • Prévention injections SQL : Utilisation systématique de PreparedStatement
  • Session utilisateur : Stockage du rôle et de l'identifiant en mémoire pendant la session

🛠️ Technologies utilisées

Java 8 MySQL 8 JDBC BCrypt Maven CommandLineMenus Git

📚 Ce que j'ai appris

  • Programmation Orientée Objet : Travail sur un code existant, compréhension du patron Singleton et du pattern Passerelle
  • Conception de bases de données : Modélisation MCD → MLD → SQL, contraintes d'intégrité, relations entre tables
  • JDBC & requêtes SQL : Connexion, PreparedStatement, ResultSet, gestion des exceptions SQL
  • Sécurité applicative : Hashage BCrypt, contrôle d'accès par rôles, prévention des injections SQL
  • Architecture logicielle : Compréhension et respect d'une architecture 3-tiers en environnement de production
  • Travail sur code existant : Lecture, compréhension et extension d'un projet Java fourni, sans en briser l'architecture
